# Security

## Pinned versions

See `requirements.lock` for exact tested versions. The lock file is
regenerated by:

```bash
uv pip compile pyproject.toml -o requirements.lock
```

(or `pip-tools`).

## Vulnerability scanning

`pip-audit` is the recommended way to check for known CVEs in the
runtime dependency tree. Run it locally before any release:

```bash
# Install pip-audit (one-off, not a runtime dep)
pip install pip-audit

# Scan the active venv
pip-audit

# Scan a specific requirements file
pip-audit -r requirements.lock

# Only show vulnerabilities with a fix available
pip-audit --strict
```

The full output (with CVE IDs) is the source of truth for "is
this dependency safe?" decisions. When `pip-audit` reports a
finding, the action is:

  1. **upgrade the dep** in `pyproject.toml` to the next safe
     version
  2. **re-run the eval suite** to confirm nothing broke
  3. **re-run this doc** to record the new lock

In CI, `pip-audit` runs on every push via
`.github/workflows/ci.yml` (see that file for the exact command).
A finding that exits non-zero **fails the build** — a deliberate
choice to keep known-vulnerable deps out of main.

## When this doc is wrong

- If `pip-audit` is uninstallable in the current Python, fall
  back to `pip list --format=json | safety check --stdin` (the
  PyUp safety DB; less current than OSV but better than nothing).
- If a CVE has **no fix yet** and we cannot upgrade, we pin
  the dep to the latest safe version (the highest version
  before the vulnerability was introduced) and document the
  reason in the commit message.
